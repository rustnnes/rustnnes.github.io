Today I Learned - Heurísticas

As Heurísticas aplicadas a teste de software dão o norte para a hora de executar os testes na aplicação.

No Intensivão do Júlio de Lima, ele fala sobre heurísticas, mais especificamente a VADER (Verbs, Auth[NZ], Data, Errors, Responsiveness). Ou seja, você sabe que vai precisar testar:

  • se os verbos foram usados e implementados corretamente;
  • se autenticação e autorização funcionam como deve. Adicione a isso negativas de acesso quando assim for o caso;
  • se o formato de dados está conforme a necessidade da aplicação que o recebe. Isso já permite projetar testes de contrato, hava vista que cada aplicação pode ter necessidade especificas de dados;
  • se erros são lançados conforme a necessidade da aplicação que o recebe;
  • se critérios de responsividade são satisfeitos.

No entanto, a impressão que tenho é que as heurísticas acabam por camuflar o tamanho dos testes que devem ser feitos, quando se trata de uma API. Naturalmente, não se testará todos os endpoints numa mesma sessão de QA, por isso a necessidade de iniciar automação de APIs o mais cedo possível, e deixar esses testes a cargo dessa automação.

Percebo que mensurar a quantidade de testes a se executar, quando isso ocorra de forma manual, também é importante. Mas para uma API com muitos endpoints, a Heurística nos diz apenas o que fazer no teste, mas não ajuda a estimar o tempo para isso.
